## 沟通模式

遇到需要用户决策的问题时，不能只抛出问题等用户回答。必须：

1. **列出可选方案**：给出所有合理选项
2. **多维度分析**：每个方案的优劣、适用场景
3. **给出推荐**：基于项目实际情况推荐一个方案，附带理由

禁止只问"A 还是 B？"而不给分析和推荐。

## 工作原则

1. **基于事实和逻辑**：所有决策和实现必须基于已验证的事实，不得基于猜测或假设
2. **遇到不确定时提问**：对需求、技术细节有疑问时，必须向用户确认，不得自行假设
3. **禁止未经验证的工作**：不在未证实的信息基础上开展工作
4. **方案必须经过 review**：任何实现方案都必须先向用户汇报并获得确认后才能开始编码。无例外
5. **查阅官方文档优先**：编写任何配置文件或集成第三方服务前，必须先查阅官方文档确认正确格式，不得凭记忆猜测
6. **证明正确性**：配置文件改动必须先展示官方文档来源，证明配置正确，再执行改动。禁止先改后查
7. **使用项目本地命令**：优先使用项目 `package.json` 中定义的脚本，而不是直接调用工具（避免版本不一致）

## 技术决策规范

多方案时必须多维度对比后决策：

| 维度 | 说明 |
|------|------|
| **代码复杂度** | 改动量、理解难度 |
| **可维护性** | 逻辑是否集中、是否易于修改 |
| **性能影响** | 运行时开销 |
| **扩展性** | 是否易于支持未来需求 |
| **架构一致性** | 是否符合现有设计模式 |
| **实现成本** | 开发时间、测试复杂度 |

### 决策优先级

1. **正确性优先**：不能为了改动小而写出有问题的代码
2. **架构一致性**：优先保持现有抽象，避免引入特殊分支
3. **可扩展性**：考虑方案是否能复用于类似场景
4. **最小改动**（辅助因素）：在满足上述条件前提下，优先选择改动较小的方案
